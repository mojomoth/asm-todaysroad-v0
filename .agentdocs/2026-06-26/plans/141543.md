# `.harness/v0/` 하네스 + `/debate-loop` 구축 계획

## Context

이 레포(`todaysroad-v0`)는 "AI 에이전트 워크플로우" 작업용 Claude Code 하네스 레포다. 슬래시 명령은
`.claude/skills/<name>/SKILL.md`(YAML frontmatter + 마크다운 본문)로 정의되고, 에이전트는 `Task` 도구로
서브에이전트(페르소나 프롬프트)를 띄워 구동한다. gstack(plan-* 리뷰 스킬)과 mattpocock(to-prd, grilling,
domain-modeling 등) 스킬이 설치돼 있어 리뷰어 역할(CEO/Eng/Design/DevEx/PM/Domain)의 출처가 된다.

요청: `.harness/v0/`에 **가이드 + 가드레일 + 랄프 루프**(루프 엔지니어링)를 갖춘 하네스를 구성하고,
입력된 기획을 A2A 에이전트 토론으로 검증하는 `/debate-loop` 명령을 만든다. moderator(중재·진행)와
devil(적대적 관찰자)이 핵심이며, 매 라운드 끝에 devil이 문제점을 제시 → moderator가 그 문제점으로 다음
라운드를 시작한다. **3회 이상** 반복 후 최종 **SPEC.md / PRD.md / DEBATE.md**를 출력한다.

확정된 설계 결정:
- 산출물 위치: **`.harness/v0/output/<slug>/`** (입력 기획별 slug 하위)
- 패널: **풀 6인** — CEO · Eng · Design · DevEx · PM · Domain + moderator + devil
- 루프 방식: **단일 세션 오케스트레이션** — `/debate-loop` 실행 시 메인 에이전트가 `Task`로 패널/devil/
  moderator 서브에이전트를 라운드마다 띄워 3회+ 토론 후 산출물 작성

## 각 파일 핵심 내용

### `.harness/v0/GUIDE.md`
- 하네스란 무엇인가: 에이전트 작업을 **가이드(무엇을·어떻게)** 와 **가드레일(하지 말 것)** 로 감싸
  결과 품질을 보장하는 골격.
- 루프 엔지니어링 개념: 한 번에 완성하지 않고 **생성 → 비판 → 재생성**을 수렴까지 반복.
- 진입점: `/debate-loop <기획 텍스트 또는 파일경로>` 사용법, 출력물 3종 설명.
- 에이전트 출처 매핑표(역할 ↔ gstack/mattpocock 스킬).

### `.harness/v0/GUARDRAILS.md`
- **출처 정직성**: 코드/사실을 지어내지 않음, 미검증은 "미검증"으로 표기(gstack ETHOS의 "프루브 잇" 정신).
- **출력 규율**: 각 에이전트는 정해진 스키마(역할별 섹션)로만 응답, 장황한 옵션 나열 금지.
- **플랜 모드 안전**: 토론 중 파일 변경 금지(읽기/탐색만), 산출물 쓰기는 루프 종료 후 `output/`에 한정.
- **루프 종료 조건**: 최소 3라운드, devil의 미해결 치명 이슈가 0이 되거나 라운드별 신규 이슈가
  없을 때(수렴) 종료. 최대 라운드 상한(예: 6) 명시해 무한 루프 방지.
- **한글 출력**(CLAUDE.md 규칙 준수).

### `.harness/v0/ralph-loop.md`
- 랄프 루프 정의: 동일 목표를 향해 같은 루프를 수렴까지 반복하는 패턴(생성자-비평가 루프).
- debate-loop에서의 구체 적용: `라운드 N = [패널 의견 → devil 비판 → moderator 종합·쟁점 도출]`,
  다음 라운드는 직전 라운드의 미해결 쟁점을 입력으로 받음.
- 수렴 판정 알고리즘 의사코드(미해결 critical 이슈 수, 신규 이슈 델타).

### `.harness/v0/agents/*.md`
- 각 페르소나 파일: frontmatter(name/role/source-skill) + 시스템 프롬프트(역할·관점·출력 스키마).
- moderator: 패널 의견 요약 → 합의/대립 정리 → devil 이슈를 의제로 다음 라운드 시작, 수렴 판정.
- devil: 입력 기획과 라운드 산출물을 **적대적으로** 검증, 치명/중대/경미로 등급화한 문제점 리스트 산출.
- 6 패널: 대응하는 gstack/mattpocock 스킬의 관점을 압축한 1라운드용 리뷰 프롬프트. 실제 스킬 파일
  경로를 "참고 출처"로 명시(예: `.claude/skills/gstack/plan-ceo-review/`).

### `.harness/v0/templates/*`
- SPEC.template.md: 목표 / 범위 / 기능 요구 / 비기능 요구 / 인터페이스·시임 / 수용 기준 / Out of scope.
- PRD.template.md: mattpocock to-prd 템플릿(Problem/Solution/User Stories/Implementation·Testing
  Decisions/Out of Scope) 차용.
- DEBATE.template.md: 라운드별 [패널 발언 / devil 문제점 / moderator 종합·쟁점] + 최종 수렴 요약.

### `.claude/skills/debate-loop/SKILL.md` (오케스트레이터)
frontmatter: `name: debate-loop`, `description`, `argument-hint: "<기획 텍스트 또는 파일 경로>"`,
`triggers: [debate-loop, 토론 루프]`, `allowed-tools: [Task, Read, Write, Bash, AskUserQuestion]`.

본문 절차(한글 지시):
1. **입력 수집**: 인자(기획 텍스트/파일경로) 파싱. 없으면 AskUserQuestion으로 기획 요청. slug 생성.
2. **하네스 로드**: `.harness/v0/GUARDRAILS.md`, `ralph-loop.md`, `agents/*` 페르소나 읽기.
3. **랄프 루프 (round 1..N, N≥3)**:
   a. 패널 6인을 `Task`로 **병렬** 호출 — 각 페르소나 프롬프트 + (직전 라운드 쟁점) 전달.
   b. devil을 `Task`로 호출 — 기획 + 패널 의견을 적대적 검증, 등급화 문제점 산출.
   c. moderator를 `Task`로 호출 — 종합·합의/대립 정리, 다음 라운드 의제(미해결 쟁점) 도출, 수렴 판정.
   d. 라운드 기록을 메모리에 누적. 수렴(미해결 critical 0 또는 신규 이슈 없음) & round≥3이면 종료,
      아니면 다음 라운드. 상한 6 도달 시 강제 종료.
4. **산출물 작성**: `.harness/v0/output/<slug>/`에 템플릿 기반으로 `DEBATE.md`(전 라운드 기록),
   `SPEC.md`, `PRD.md` 작성.
5. **요약 보고**: 라운드 수, 수렴 여부, 미해결 잔여 리스크, 산출물 경로를 한글로 보고.

## 재사용 (기존 자산 활용)
- 패널 페르소나는 **새 코드가 아니라** 기존 gstack `.claude/skills/gstack/plan-*-review/`와 mattpocock
  `.claude/skills/{to-prd,domain-modeling,grilling}/SKILL.md`의 관점을 출처로 인용·압축한다.
- PRD 출력은 `to-prd/SKILL.md`의 `<prd-template>`을 그대로 차용.
- 명령 정의 형식·frontmatter 키는 기존 스킬(`handoff/SKILL.md`, `gstack/SKILL.md`) 컨벤션을 따른다.
- A2A는 별도 인프라 없이 Claude Code `Task` 서브에이전트로 구현(단일 세션 오케스트레이션).

## 검증 방법
1. 파일 생성 후 트리 확인: `.harness/v0/` 및 `.claude/skills/debate-loop/SKILL.md` 존재.
2. 짧은 샘플 기획으로 `/debate-loop` 실행 → 3라운드 이상 수행되고 `output/<slug>/`에 SPEC.md, PRD.md,
   DEBATE.md 3종이 생성되는지 확인.
3. DEBATE.md에 라운드별 devil 문제점 → moderator 의제 인계 흐름이 보이는지, GUARDRAILS의 수렴/한글
   규칙이 지켜졌는지 점검.
4. SKILL.md frontmatter가 다른 스킬처럼 정상 파싱되어 `/debate-loop`로 노출되는지 확인.
