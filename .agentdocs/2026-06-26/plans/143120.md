# /develop-loop — 개발 루프 명령어 생성 계획

## Context (배경)

사용자는 `.harness/v0/`에 이미 존재하는 멀티에이전트 하네스(GUARDRAILS·ralph-loop·agents·templates)와
`debate-loop` 스킬 패턴을 기반으로, **실제 개발을 수행하는 랄프 루프** 명령어 `/develop-loop`를 원한다.

요구된 파이프라인:
```
Eval (가드레일 동적 생성: Hook + GUARDRAILS.md)
  ↓
PM (기획 → 작업 분해)
  ↓
Orchestrator (작업 분배)
  ↓
FrontEnd — BackEnd — UI Art   (FE/UI Art 는 반드시 supanova 스킬 사용)
  ↓
QA (검증)
  ↓
Eval (판단 → 다시 Loop or Goal)
```

debate-loop는 **읽기 전용 토론**이지만, develop-loop는 **실제 코드를 쓰는** 개발 루프라는 점이 핵심 차이다.

### 확정된 결정 (사용자 응답)
- **supanova**: 로컬 미설치 → 루프 0단계에서 `git clone https://github.com/uxjoseph/supanova-design-skill ~/.claude/skills/supanova` 설치 단계 추가, 실패 시 안내.
- **쓰기 범위**: git 워크트리/브랜치 격리. FE/BE/UI Art 에이전트는 격리 작업공간에서 코드 작성, QA·Eval 통과 후 통합.
- **Hook 적용**: 별도 하네스 폴더 `.harness/v0/develop-loop/`에 GUARDRAILS.md·hook 생성. 기존 `.claude/settings.json`은 미변경, 루프 내부 검증용으로 참조.

### ⚠️ 선행 제약
프로젝트가 **git 레포가 아님**(`git init` 안 됨). 워크트리 격리를 쓰려면 루프 0단계에서 `git init` +
초기 커밋이 필요. develop-loop이 이를 자동 수행하되 사용자에게 1회 고지.

## 생성할 파일

### 1. 스킬 본체
`/Users/jyl/work/repo/todaysroad-v0/.claude/skills/develop-loop/SKILL.md`
- frontmatter: `name: develop-loop`, 한글 description, `argument-hint`, `triggers: [develop-loop, 개발 루프, 디벨롭 루프]`,
  `allowed-tools: [Task, Read, Write, Edit, Bash, AskUserQuestion, Skill]`
- debate-loop/SKILL.md 형식을 미러링. 단계 구성:
  - **0. 하네스 로드 + 환경 준비**: 아래 하네스 파일 Read, supanova 설치 확인/clone, `git init`(필요 시),
    slug 생성, 산출 폴더 `mkdir -p .harness/v0/develop-loop/output/<slug>`.
  - **1. 입력 수집**: `$ARGUMENTS`(텍스트/파일경로) 또는 AskUserQuestion으로 기획 확보.
  - **2. Eval — 가드레일 동적 생성 (랄프 루프)**: Eval 에이전트가 입력 컨텍스트를 분석해
    `.harness/v0/develop-loop/output/<slug>/GUARDRAILS.md` + hook 스크립트(`hooks/*.sh`)를 생성.
    랄프 루프로 가드레일 자체를 수렴시킴(생성→자기비판→정제).
  - **3. PM**: 확정 가드레일 하에 기획을 작업 단위(task)로 분해, 수용 기준 명시.
  - **4. Orchestrator**: task를 FE/BE/UI Art 트랙으로 분배, 의존성 정렬.
  - **5. 구현 (병렬, 워크트리 격리)**: `Task` 3트랙 병렬. FE·UI Art 프롬프트에 **`/supanova` 스킬 사용 필수** 명시.
    각 에이전트는 `isolation: worktree` 격리 작업공간에서 코드 작성.
  - **6. QA**: 구현 결과를 테스트·검증(빌드/린트/테스트 실행, 수용 기준 대조).
  - **7. Eval — 판정**: QA 결과 + 동적 GUARDRAILS 대조해 `CONVERGED / CONTINUE / FORCE_STOP` 판정.
    미충족이면 미해결 이슈를 다음 라운드 Orchestrator 입력으로 되먹임.
  - **8. 루프 제어**: `round >= 3` & 수렴 → 종료. `round == 6` 강제 종료(잔여 리스크 명시). 무한 루프 금지.
  - **9. 통합 & 보고**: 통과 시 워크트리 변경을 메인 브랜치로 병합, 산출물·잔여 리스크 한글 보고.

## 검증 (How to verify)
1. **로드 확인**: 새 세션에서 `/develop-loop` 입력 → SKILL.md frontmatter가 파싱되어 트리거되는지.
2. **드라이런**: 작은 기획(예: "랜딩 페이지 헤더 추가")을 인자로 주고 0~2단계만 관찰 —
   supanova 설치 시도, `git init`(필요시), Eval이 `output/<slug>/GUARDRAILS.md` + hook 생성하는지 확인.
3. **루프 종료**: round 카운터가 6에서 강제 종료되고 잔여 리스크가 산출물에 기재되는지(무한 루프 없음).
4. **격리 확인**: 구현 에이전트가 워크트리에서 작업하고 메인 트리가 QA 통과 전까지 오염되지 않는지 `git status`로 확인.
5. **supanova 강제**: FE/UI Art 에이전트 프롬프트에 `/supanova` 사용 지시가 포함됐는지 SKILL.md 검토.
