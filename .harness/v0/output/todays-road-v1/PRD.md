# PRD — 오늘의길 (Today's Road)

> 생성: `/debate-loop` · slug: `todays-road-v1` · 출처 템플릿: mattpocock `to-prd`

## Problem Statement
도시 거주자와 여행객은 "지금, 여기서, 이 정도 시간 동안, 이런 기분으로" 걷고 싶을 때
적합한 산책길을 빠르게 찾기 어렵다. 늘 같은 길은 지겹고, 새 길을 찾으려면 과도하게 탐색해야 한다.
기존 길찾기 앱은 목적지를 전제하며, 산책 중에도 계속 폰 화면을 봐야 해서 산책 경험을 해친다.

## Solution
현재 위치 + 가용 시간 + 감성 태그를 받아, **강남역 반경 1.5km 내 검증된 산책 코스를 항상 5개 추천**한다.
사용자는 코스를 골라 산책을 시작하고, **음성 턴안내 + 단순 시각모드**로 폰을 덜 보면서 완주한다.
경로상 숨은 카페·식당 3곳도 함께 발견한다.
MVP 한 줄 가치: **"폰을 덜 보면서 걷기 좋은 길 속 숨은 카페·식당을 발견하는 산책 가이드."**

## User Stories
1. As a 강남 거주자, I want 현위치에서 30분 산책 코스 5개를 즉시 추천받기, so that 끝없이 스크롤하지 않고 새 길을 고른다.
2. As a 여행객, I want 강남역 근처에서 시간·기분에 맞는 코스를 받기, so that 낯선 동네에서 빠르게 걸을 곳을 찾는다.
3. As a 폰 보기 싫은 산책자, I want 음성 턴안내로 길을 안내받기, so that 폰을 주머니에 넣고 걷는다.
4. As a 분위기 중심 사용자, I want "조용한/강변/카페 많은" 같은 감성 태그로 고르기, so that 기분에 맞는 길을 선택한다.
5. As a 시간 제약 사용자, I want 15/30/45/60/90분 단위로 필터링, so that 정확히 내 시간에 맞는 코스를 고른다.
6. As a 신뢰가 필요한 사용자, I want 거리·걸음수·예상시간을 코스 카드에서 확인, so that 데이터가 실제임을 믿는다.
7. As a 카페 애호가, I want 경로상 대표 카페/식당 3곳을 보기, so that 산책 중·후 쉴 곳을 안다.
8. As a 첫 사용자, I want 위치 권한 없이도 강남역 디폴트 추천을 보기, so that 진입 장벽 없이 가치를 체험한다.
9. As a GPS 약한 구간의 사용자, I want 음성 끊김 시 단순 시각모드로 전환, so that 길을 잃지 않는다.
10. As a 잘못된 정보를 본 사용자, I want POI 오류를 신고하기, so that 폐업한 카페가 빠르게 숨겨진다.

## Implementation Decisions
- **추천 엔진**: 하드필터(현위치 반경) → 소프트 랭킹(시간 근접 점수 + 태그 매칭 자카드×객관축 1.5배) → 항상 상위 K=5 노출. 결과 0개 금지. 반경 fallback 1→2→3→5→10km, 이후 태그 완화.
- **코스 엔티티(단일 스키마)**: `{ id, name, type: loop|a-b, source: curated, pathCoordinates: LatLng[], metrics{distanceKm, estimatedMin, stepCount, elevationGainM}, sentimentTags[], pois[3] }`. 메트릭은 경로좌표서 결정론적 재계산.
  - 예상시간(분) = 거리/속도×60 + 고도보정. 속도: 평탄 4.5 / 상향 3 / 하향 5 km/h(경사도 임계 5%). 고도보정: 상승 100m당 +6분, 하강 100m당 -3분. 걸음수 = 거리(m)/0.68m. (모두 잠정값, 베타 실측 ±15% 보정)
- **POI**: 카카오 실시간 API 미사용. 코스당 대표 카페/식당 3개(경로 1/3·2/3·종료, 50m내)를 큐레이션 메타데이터로 수작업. 월 1회 갱신 + 신고 버튼.
- **음성/네비**: MVP=음성on 턴안내(산책 시작 시 1회 프롬프트, 기본 ON) + 단순 시각모드(큰 화살표 + 남은거리). 햅틱 2패턴(안내/도착). 사전생성 MP3 캐시 재생, 맵매칭 GPS ±10m 보정. 본격 햅틱 다패턴 + earcon은 사용성테스트(5인) 통과를 V1 게이트로.
- **태그 온톨로지(하이브리드)**: 객관축(거리 버킷/유형/시간대) + 주관 감성축. 배타쌍(조용↔활기), 공존 가능(강변+조용). 관리자 2인 검수(합의율 ≥90%).
- **아키텍처**: 5모듈 — Course / Navigation / Voice / POI / MapAdapter. 최상위 시임 `Navigation.onLocationUpdate(lat, lon, accuracy)`. 지도·POI·TTS는 인터페이스로 추상화(교체 가능).
- **권역**: 강남역 반경 1.5km, 5개 마이크로권역. 초기 코스 10~12개(파일럿 3개 → 1개월 15개 → 3개월 100개 → 6개월 500개 로드맵).

## Testing Decisions
- 외부 동작만 테스트. 추천 엔진(반경 필터·태그 매칭·랭킹)을 순수 함수로 격리해 단위 테스트(필터 조합 5+케이스, fallback, 항상 상위 K개).
- 지도/TTS는 mock(SilentTTS, MockMapAdapter)로 치환, 실제 카카오 SDK는 스테이징/수동 통합 테스트.
- E2E: `onLocationUpdate` 시뮬레이션으로 위치→추천→경로표시→음성/시각 안내 전체 흐름.
- 현장검증: 강남역 도시캐니언 5회 왕복, 턴 감지 성공률 ≥90%, 발화 타이밍·MP3 지연 측정.
- 게이트 측정: hit rate(코스당 3회 왕복 도달성공률), 사용성테스트(5인 현지) 정량 기준.

## Out of Scope
- 카카오 실시간 POI 조회, 루프 코스 자동 생성, UGC/리뷰/소셜, 다국어, 강남 외 지역, 실시간 혼잡도/날씨, 본격 햅틱+earcon, 위치 히스토리 분석(정책 미정).

## Further Notes — 출시 전 필수 잔여 리스크 (FORCE_STOP, devil 최종)
이 PRD의 설계 결정은 토론으로 수렴했으나, 다음은 **실제 실행해야만 닫히는 V1 출시 게이트 항목**이다.
- **R6-2 TTS 미선택(우선순위 최상)**: 음성이 핵심 경험인데 엔진/라이선스/월 비용 미정. 이를 정해야 현장검증·사용성테스트가 가능. Google TTS vs Naver Clova vs 사전녹음 비교 필요.
- **R6-1 게이트 실행계획**: 검증 체크리스트(.md), 담당자, 일정, 출시일, 롤백 기준 미확정(2주 내).
- **R6-3 현장검증 일반화**: 강남역 1지역·여름·대낮 단일 조건. 겨울/우천/야간/타지역 재검증 로드맵 필요.
- **R6-5 약관 법무**: 면책조항·개인정보(위치) 처리방침의 한국 법(전자상거래법·PIPA) 실효성 법무 자문 필수.
- **R6-6 성능**: 단일 콜백 메인스레드 부하 저사양 실기기 측정.
- **R6-9/R6-10**: GPS 약신호 시각모드 이탈률 측정, 위치 히스토리 저장/개인정보 정책 명시.
- **R1-10 접근성**: 현재 5/10. 음성 스크린리더 호환·WCAG AA 핵심 항목은 V1.
- **R5-4 스케일**: 큐레이션 확장 로드맵은 리소스·자동화·지형데이터 미검증. 파일럿 3코스 후 재산정.
