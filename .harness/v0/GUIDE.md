# GUIDE — 하네스 가이드

## 1. 하네스란

하네스는 에이전트가 자유롭게 헤매지 않도록 작업을 **골격**으로 감싸는 장치다. 두 축으로 이뤄진다.

- **가이드** — 무엇을, 어떤 순서로, 어떤 관점으로 할지. (이 문서 + `agents/*`)
- **가드레일** — 무엇을 하면 안 되는지, 어디서 멈춰야 하는지. (`GUARDRAILS.md`)

여기에 **루프 엔지니어링**을 더해, 한 번의 호출로 끝내지 않고 **생성 → 비판 → 재생성**을 수렴까지
반복함으로써 품질을 끌어올린다. 반복 패턴은 `ralph-loop.md`에 정의된 **랄프 루프**를 따른다.

## 2. 루프 엔지니어링 개념

> 좋은 결과는 한 번에 나오지 않는다. 좋은 **루프**에서 나온다.

- **생성자(Generators)** — 6명의 전문가 패널이 각자 관점에서 기획을 평가·보강한다.
- **비평자(Critic)** — devil(적대적 관찰자)이 기획과 패널 산출물의 허점을 등급화해 들춘다.
- **중재자(Moderator)** — 합의/대립을 정리하고, devil이 던진 미해결 쟁점을 **다음 라운드의 의제**로
  넘긴다. 즉 한 라운드의 출력이 다음 라운드의 입력이 된다.
- **수렴** — 미해결 치명(critical) 이슈가 0이 되거나 라운드 간 신규 이슈가 없어질 때까지 반복한다.
  단 최소 3라운드, 최대 6라운드.

## 3. 사용법

```
/debate-loop <기획 텍스트 또는 파일 경로>
```

- 인자로 기획을 직접 적거나 파일 경로(예: `docs/idea.md`)를 넘긴다.
- 인자가 없으면 명령이 기획을 물어본다.
- 종료 시 `.harness/v0/output/<slug>/`에 `SPEC.md`, `PRD.md`, `DEBATE.md`가 생성된다.

## 4. 에이전트 출처 매핑

패널 페르소나는 새로 발명한 것이 아니라 이미 설치된 gstack/mattpocock 스킬의 관점을 압축한 것이다.

| 에이전트 | 역할 | 출처 스킬 |
|----------|------|-----------|
| `ceo` | 전략 · 스코프 · 10-star 제품 | `.claude/skills/gstack/plan-ceo-review/` |
| `eng` | 아키텍처 · 데이터흐름 · 엣지케이스 · 테스트 | `.claude/skills/gstack/plan-eng-review/` |
| `design` | 디자인 차원 0–10 평가 | `.claude/skills/gstack/plan-design-review/` |
| `devex` | TTHW · 마법의 순간 · 마찰점 | `.claude/skills/gstack/plan-devex-review/` |
| `pm` | 문제정의 · 유저스토리 · 스코프 | `.claude/skills/to-prd/` |
| `domain` | 도메인 용어 · 일관성 · ADR | `.claude/skills/domain-modeling/` |
| `devil` | 적대적 비판 (그릴링 정신) | `.claude/skills/grilling/` |
| `moderator` | 회의 진행 · 합의 도출 · 수렴 판정 | (하네스 고유) |

## 5. 출력물 3종

- **SPEC.md** — 토론을 거쳐 확정된 실행 가능한 스펙(목표·범위·요구사항·수용기준).
- **PRD.md** — `to-prd` 템플릿 기반 제품 요구 문서(문제·해결·유저스토리·결정·스코프).
- **DEBATE.md** — 라운드별 [패널 발언 → devil 문제점 → moderator 종합·쟁점] 전체 기록 + 수렴 요약.
