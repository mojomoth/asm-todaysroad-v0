# GUARDRAILS — 하네스 가드레일

토론에 참여하는 모든 에이전트(패널 · devil · moderator)와 오케스트레이터가 반드시 지켜야 하는 규칙.

## G1. 출처 정직성 (Prove it)

- 코드·사실·수치를 **지어내지 않는다.** 확인되지 않은 주장은 `미검증`으로 명시한다.
- 레포 사실이 필요하면 추측하지 말고 탐색(Read/Grep)한다.
- "이건 잘 될 것"이라는 낙관은 증거 없이는 쓰지 않는다. (gstack ETHOS의 정신)

## G2. 출력 규율

- 각 에이전트는 자신의 페르소나 파일이 지정한 **출력 스키마(섹션)** 로만 응답한다.
- 장황한 옵션 나열·반복 금지. 추천이 있으면 추천을 제시하고 근거를 짧게 단다.
- devil은 모든 문제점을 **등급화**한다: `critical`(치명) / `major`(중대) / `minor`(경미).
- 각 문제점에는 고유 ID(`R{라운드}-{순번}`)를 부여해 다음 라운드에서 추적 가능하게 한다.

## G3. 플랜·토론 단계 안전

- 토론(라운드) 진행 중에는 **파일을 변경하지 않는다.** 읽기·탐색만 허용.
- 산출물 쓰기는 **루프가 종료된 후**에만, **`.harness/v0/output/<slug>/`** 안에서만 수행한다.
- 그 외 경로(레포 코드, 다른 하네스 파일)는 절대 수정하지 않는다.
- 파괴적 명령(rm -rf, force-push 등) 금지.

## G4. 루프 종료 조건

루프는 다음을 **모두** 만족할 때 종료한다.

1. **최소 라운드 충족**: `round >= 3`.
2. **수렴**: 다음 중 하나
   - devil이 제기한 미해결 `critical` 이슈 수 = 0, **또는**
   - 직전 라운드 대비 **신규 이슈가 없음**(델타 = 0).

수렴하지 못해도 **`round == 6`(상한)** 에 도달하면 강제 종료하고, 미해결 잔여 리스크를 산출물에 명시한다.
무한 루프는 절대 허용하지 않는다.

## G5. 라운드 인계 규칙

- 매 라운드 끝에서 devil이 문제점을 제시하고, moderator가 그 문제점을 **다음 라운드 의제**로 정리한다.
- 다음 라운드 패널은 직전 라운드의 미해결 쟁점을 **입력으로 받아** 자신 관점에서 해소안을 제시한다.
- 해소된 이슈는 `resolved`로, 새로 발견된 이슈는 새 ID로 기록한다.

## G6. 언어

- 모든 산출물과 에이전트 응답은 **한글**로 작성한다. (프로젝트 `CLAUDE.md` 규칙)
- 고유명사·코드 식별자·기술 용어는 원문 유지 가능.
